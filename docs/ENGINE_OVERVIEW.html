Sama Engine

A cross-platform C++20 game engine targeting Windows, macOS, iOS and Android. Built on bgfx, Jolt and SoLoud. Interview prep document — read top to bottom for the talking track, or jump straight to a section.

1. The initial proposal

The 30-second answer. I wrote down the brief myself before any code existed: a custom, easy-to-use C++ engine and editor that runs the same game on Windows, Mac, iOS and Android, with an ECS data layout, Metal on Apple and Vulkan elsewhere, multi-threaded with per-platform thread caps, modern rendering techniques landed step by step, 60 fps on mobile and 90–120 fps on desktop, with tests and tight binary size as non-negotiables.

The proposal lives at the top of the repo in CustomGameEngine.md — the very first thing in the working tree. I'm quoting it verbatim here so the interviewer can see what the original intent was, before I knew where the tradeoffs would land.

What survived and what didn't, in one sentence each:

  • ECS, cross-platform, C++20, Metal+Vulkan via bgfx, fps targets, ImGui, glTF-first — all survived.
  • Ray tracing — explicitly tabled long-term; bgfx doesn't expose RT APIs and forking it would dwarf the rest of the engine. Documented in docs/NOTES.md:136-140.
  • FBX — punted in favour of glTF-only via cgltf; FBX adds ~10 MB of assimp for little gain.
  • "Be deliberate about external libraries" — drove every dependency choice: rejected tinygltf and fastgltf for cgltf, rejected FMOD (~3–6 MB, royalty) for SoLoud + miniaudio, rejected nlohmann/json for rapidjson, rejected Diligent / Filament for bgfx.

2. High-level architecture

The 30-second answer. The engine is a C++20 codebase built with CMake, layered as ECS at the core, with subsystems (rendering, animation, physics, audio, assets, threading, memory, input) that all express their state as ECS components. Games implement an IGame interface and are driven by a cross-platform GameRunner with per-platform entry points (NativeActivity on Android, UIApplication on iOS, GLFW on desktop). Rendering goes through bgfx, which gives me Metal on Apple, Vulkan everywhere else, with one source of shaders.

The IGame / GameRunner pattern

Games never see main() or platform entry points — they implement engine::IGame and GameRunner drives them. The per-platform shim (NativeActivity callback on Android, UIApplicationMain on iOS, GLFW loop on desktop) calls into GameRunner. This is the load-bearing abstraction that makes "one game, four platforms" possible.

Build system

Plain CMake. External libraries via FetchContent (Jolt, glm, glfw, bgfx, Catch2, cgltf) with stb and SoLoud vendored. Key targets: engine_tests, engine_screenshot_tests, helmet_demo, hierarchy_demo, physics_demo, audio_demo, animation_demo, plus per-platform app shells (android/, ios/, sama_editor).

3. The game loop

The 30-second answer. Each frame the engine polls OS events, updates input and animation, runs physics on a fixed timestep with an accumulator (1/60 s, max 4 substeps), updates transforms, builds visibility tags + the cluster light grid, records draw calls via bgfx encoders, runs the post-process chain, and ends with bgfx::frame() which hands the command buffer to the bgfx render thread. The frame arena resets to zero allocated bytes at the end of every frame.

bgfx threading model

bgfx ships with an internal render thread by default; the user thread records commands, and bgfx::frame() is an asynchronous hand-off via a lock-free ring. The engine picks single-threaded mode only when EngineDesc::singleThreaded = true by calling bgfx::renderFrame() before bgfx::init() — that pre-init call is the documented way to inhibit the render thread. See engine/rendering/Renderer.cpp:48-65 for the comment block.

Honesty flag — what multi-threaded actually buys you. Prior versions of this doc and docs/NOTES.md claimed a ~20 ms → ~0.1 ms bgfx::frame() drop from flipping to multi-threaded. That number is real for perf_smoke-class workloads where GPU work is well under the vsync period — the ring stays drained and bgfx::frame() collapses to the hand-off cost. For a GPU-bound shipping game at 1080×2424 PBR + bloom on Pixel 9, a reporting team measured the post-flip bgfx::frame() at 15+ ms despite the threading change being correctly engaged. Mechanism: with BGFX_RESET_VSYNC set (Renderer.cpp:69 + :267), the render thread blocks on vkAcquireNextImageKHR waiting for vblank; the ring fills; the game thread's next bgfx::frame() blocks on a free ring slot. The wait migrates from the GPU-fence wait to the ring back-pressure wait — it doesn't disappear. See docs/NOTES.md "Correction (2026-06-16): expected ~0.1 ms was scene-specific" for the full mechanism and the four fix paths (reduce GPU work, target 30 fps with Surface.setFrameRate, disable vsync as a diagnostic, deeper swapchain). What multi-threaded DOES still buy: CPU/GPU parallelism (render thread issues Vulkan calls while game thread builds next frame's command buffer), and game-thread cost outside bgfx::frame() goes from 16 ms wait → 0.05 ms hand-off, freeing the other systems' budget.

Follow-up: how is the variable-step update reconciled with fixed-step physics?

The engine calls physics.step(deltaTime, kMaxSubSteps=4) in PhysicsSystem::update. Jolt manages the accumulator internally with a fixed 1/60 s timestep, running up to 4 substeps if the frame falls behind. Everything else (Animation, Transform, Render) is variable-step. This is the standard accumulator pattern; no interpolation between physics ticks is performed, which is a known limitation for very slow frames.

4. ECS architecture

The 30-second answer. Sparse-set ECS, custom, no external library. Entities are 64-bit IDs with a generation counter; components are stored in a SparseSet<T> per type (sparse index array, packed dense data array, parallel denseEntities_ array for reverse lookup); queries go through View<Components...> which picks the smallest store as the outer loop and filters by the others. Systems declare Reads and Writes as TypeLists and a constexpr buildSchedule<...>() bakes phase groupings into the binary at compile time. ~57 test cases / 3423 assertions on the ECS alone.

Sparse set in pictures

Entity indices:    0     1     2     3     4     5
                   ┌─────┬─────┬─────┬─────┬─────┬─────┐
sparse_:           │  2  │  ∅  │  0  │  1  │  ∅  │  3  │  entity index → dense index
                   └─────┴─────┴─────┴─────┴─────┴─────┘
                     ↓              ↓    ↓              ↓
                   ┌─────┬─────┬─────┬─────┐
dense_:            │ C2  │ C0  │ C3  │ C5  │  packed T (cache-friendly outer loop)
denseEntities_:    │ E2  │ E0  │ E3  │ E5  │  parallel: which entity owns each slot
                   └─────┴─────┴─────┴─────┘

Insert is O(1). Remove is swap-and-pop: copy the tail slot into the removed slot, update its sparse entry, shrink by one. Iteration over dense_ is a tight loop over a packed array — exactly what the CPU prefetcher wants.

Entity IDs with generations

An EntityID is a uint64_t packed as [generation : 32 | index : 32]. Destroying an entity increments the generation at that index, so any stale copy of the old ID fails the isValid() check. If the generation wraps to zero the index is retired permanently — that's the conservative choice over silently letting stale IDs ride.

Views and multi-component join

reg.view<Position, Velocity>().each(fn) picks the smaller of the two sparse sets as the outer loop, then probes the other for contains() on each entity. Null stores (component type never emplaced) are treated as size 0 and the view immediately yields nothing.

engine/ecs/Registry.h (typical use) 
// Movement system body — note the smallest-set outer-loop happens implicitly.
reg.view<Position, Velocity>().each([dt](EntityID e, Position& p, Velocity& v) {
    p.x += v.x * dt;
    p.y += v.y * dt;
    p.z += v.z * dt;
});

Pros / Cons (interview-ready)

ProsCons
Cache-friendly iteration: dense arrays are tight. Structural mutations during iteration (emplace/remove) are unsafe → need command buffers for parallel cull.
Zero-overhead component access through SparseSet<T>. Type-level conflict matrix is conservative: two systems that write the same component on disjoint entities still serialise.
Compile-time conflict detection — data races impossible by construction once Reads/Writes are declared. Runtime-typed component lookup goes through unordered_dense::map<type_index, ...> — one hash + virtual dispatch per access (findStore<T>).
No runtime registration — all systems known at compile time. Means no runtime scripting layer in the ECS itself — gameplay is C++.
Empty-type specialisation: VisibleTag on 50k entities costs only the 200 KB sparse index array, no dense storage. Entities aren't move-constructible in some patterns; care needed in test fixtures.
None of the built-in render systems (TransformSystem, FrustumCullSystem, DrawCallBuildSystem, ShadowCullSystem, LightClusterBuilder) actually declare Reads/Writes yet. The scheduler is real and tested, but the schedule can't reason about the built-in systems yet — wiring those declarations is per-system follow-up work, gated on per-tier device measurement. docs/THREADING_ARCHITECTURE.md says so explicitly at the top.

5a. Rendering — the big one

The 30-second answer. Forward+ clustered PBR pipeline on bgfx. The renderer is ECS-driven — there's no separate scene representation. Lighting is a 16×9×24 cluster grid with up to 256 lights packed into three GPU textures. Shadows are CSM with a hybrid log/uniform split, three cascades by default, packed into a single 2048² atlas with cascadeMask bits driving the cull. PBR is Cook-Torrance GGX + Smith + Fresnel-Schlick with split-sum IBL. Post-process is HDR → bloom (2N-1 passes, tier-aware) → ACES tonemap → FXAA. Skinning uses u_model[128] as the bone palette. Draw calls go through bgfx::Encoder so they record thread-local instead of through the global mutex.

Why bgfx

Chosen over a custom HAL, IGL, Filament, Magnum, and Diligent. The reasoning is in docs/NOTES.md:121-141:

  • Versus custom HAL: "binary size savings (~1–2 MB) do not justify months of low-level Vulkan/Metal boilerplate." With one developer on a multi-platform engine, every month spent on VkInstance + VkPhysicalDevice + VkSwapchainKHR + Metal layer plumbing is a month not spent on the actual graphics techniques.
  • Versus IGL (Meta): "bgfx has stronger game engine track record, more documentation/examples, includes shader tooling (shaderc)." IGL is noted as stronger on Android long-tail device fragmentation — flagged as revisit-if-it-becomes-a-pain-point.
  • Versus Filament/Magnum: Filament is a renderer, not a rendering abstraction — using it would have meant inheriting its scene graph, its material model, its asset pipeline. The whole point was to own those parts.
  • What bgfx gives: one shader source (.sc) → compiled to Metal MSL, SPIR-V, HLSL via shaderc. Encoder API for multi-threaded draw recording. Threading model built around a render thread + async hand-off (so bgfx::frame() doesn't block the game thread). MIT license. Mature on mobile, MoltenVK fallback on macOS unused because Metal is first-class there.

The threading model is the load-bearing detail. The default mode has bgfx spawn its own render thread. bgfx::frame() hands the command buffer off through a lock-free ring and returns immediately; the render thread does the actual vkSubmit / MTLCommandBuffer::commit. Single-threaded mode (used for tests and the Noop backend) is opted into by calling bgfx::renderFrame() before bgfx::init().

Forward clustered lighting

The frustum is sliced into a 16×9×24 cluster grid (3,456 clusters total). Cluster ID is computed as:

tile.xy   = floor(gl_FragCoord.xy / vec2(screenW/16, screenH/9));
sliceIdx  = int(log(viewZ / nearPlane) / log(farPlane/nearPlane) * 24);
clusterIdx = sliceIdx * 144 + tile.y * 16 + tile.x;   // 144 = 16*9

Logarithmic depth slicing concentrates clusters near the camera (where small lights matter most) without hand-tuning the per-slice depth bounds.

Light assignment to clusters happens CPU-side in LightClusterBuilder: for each cluster, build an AABB by unprojecting tile corners + near/far slice planes, then sphere-vs-AABB test every light. Two-pass: count, then prefix-sum offsets, then fill.

GPU layout — three textures, because uniform arrays can't hold 256 lights × 4 vec4s (bgfx's BGFX_CONFIG_MAX_UNIFORMS=512 caps that at 512 vec4s total):

Why forward + clustered, not deferred? Three reasons, all mobile-driven:

  • TBDR-friendly. Apple and Mali / Adreno are tile-based deferred renderers. A G-buffer (4–5 fullscreen RTs) flushes tile memory on every store, costing main-memory bandwidth that's the actual bottleneck on mobile.
  • Transparency works naturally. Deferred needs a parallel forward path for translucents anyway — clustered just runs the same lighting loop.
  • Lower memory footprint. No G-buffer textures.

Cascaded shadow maps (CSM)

Three cascades by default (configurable up to 4 via RenderSettings). Splits via a hybrid log/uniform scheme — the Nvidia GPU Gems 3 formula:

C_i = lambda * C_log_i + (1 - lambda) * C_uni_i

Code in engine/rendering/CsmSplitCalculator.cpp. lambda=0 → linear (equal-size slices), lambda=1 → logarithmic (geometric growth). Useful default is around 0.5.

Per cascade, cascadeLightProj() fits a tight orthographic projection around the camera sub-frustum slice transformed into light space. The shadow atlas packs all cascades into a single texture (2048² with four 1024×2048 columns for 4 cascades, or 4096² for 2048-res cascades) — one SAMPLER2DSHADOW in the shader instead of one sampler per cascade.

Single-pass multi-cascade culling. ShadowCullSystem::update(reg, res, span<Frustum>, baseCascadeIdx) walks the mesh view once, builds the world AABB once per entity, tests against every frustum in the span and accumulates the result as a bit mask in ShadowVisibleTag::cascadeMask. Was 3× iteration before; this was the audit win (#cull-shadow-multi) — about 66% less shadow-cull work at 3 cascades.

Sampling in fs_pbr.sc uses PCF 2×2 with slope-scaled bias clamped to [0.002, 0.02]:

shadow += shadow2D(s_shadowMap, vec3(shadowCoord.xy + vec2( texelSize, -texelSize), shadowZ));
shadow += shadow2D(s_shadowMap, vec3(shadowCoord.xy + vec2(-texelSize,  texelSize), shadowZ));
shadow += shadow2D(s_shadowMap, vec3(shadowCoord.xy + vec2( texelSize,  texelSize), shadowZ));
shadow *= 0.25;
No cascade-transition blending — there's a visible seam where one cascade hands off to the next. It's a known follow-up; the fix is to sample both cascades in a transition zone and lerp.

PBR (Cook-Torrance + IBL)

BRDF: the standard Cook-Torrance microfacet model.

  • D (normal distribution): GGX / Trowbridge-Reitz (distributionGGX at fs_pbr.sc:78-86).
  • G (geometry): Smith with Schlick-GGX (geometrySchlick + geometrySmith).
  • F (Fresnel): Schlick approximation (fresnelSchlick).

Energy conservation: the diffuse term is multiplied by kD = (1 - F) * (1 - metallic) — metals have no diffuse, dielectrics attenuate diffuse by the part of incoming light not reflected specularly.

Material textures, slot map:

Tangent-space normal mapping: TBN is reconstructed from the interpolated tangent + bitangent + geometry normal. The transform is expanded manually (T*ns.x + B*ns.y + Ngeom*ns.z) instead of using mtxFromCols because Metal's shader path transposes on mtxFromCols.

IBL — split-sum approximation (Karis SIGGRAPH '13):

  • Irradiance cubemap (RGBA16F) — pre-convolved diffuse, sampled by surface normal.
  • Prefiltered specular cubemap (RGBA16F, mip chain) — sampled by reflection vector, LOD = roughness * kMaxMipLevels.
  • BRDF integration LUT (RG16F) — 2D LUT indexed by (NdotV, roughness).

mediump precision policy. One of the audit wins (#S1 precision half) was annotating fs_pbr.sc with per-variable mediump instead of a blanket precision mediump float. The policy is documented at the top of the shader and is interview-relevant:

  • mediump: BRDF helpers (D/G/F), 8-bit-texture samples, light colour, cone factors, IBL samples, accumulators (Lo / ambient / emissive). Bounded values that fit in fp16.
  • highp: world/view-space positions, camera position, shadow UVs, cluster index math (fp16 stores integers exactly only up to 2048; the flat light index can reach 8191), and the light-position-in-view-space row (because distance() against radius needs to be accurate at the radius boundary).

On Mali Bifrost / Valhall the ESSL mediump keyword routes to fp16 ALU + bandwidth; on SPIR-V it emits RelaxedPrecision which SPIRV-Cross translates to half in MSL. Desktop GLSL ignores the keyword. Net effect: real fp16 paths on the mobile targets that matter, no desktop regression.

Draw call submission

This is the headline rendering optimisation (#R1). DrawCallBuildSystem opens a bgfx::Encoder once per system call (bgfx::begin()), then every setTransform / setTexture / submit runs thread-local (~50 ns) instead of grabbing the global bgfx command-list mutex (~1–2 μs on Pixel-5a-class CPUs).

Measured on the Android emulator (702-entity perf_smoke, 600 frames, A/B same session):

  • Mean: 0.043 → 0.042 ms (flat within noise — uncontested mutex is fast)
  • p99: 0.167 → 0.116 ms (−30%)
  • Max: 0.348 → 0.174 ms (−50%)

The tail-latency drop is the signature of removed mutex contention — mean is flat because uncontested grabs are fast, p99/max capture the times another thread held it. On real ARM hardware the mean would also drop because the emulator's x86_64 translation has lower native mutex cost than the target.

Follow-up: didn't the audit originally suggest hoisting frame-constant uniforms?

Yes — and the investigation found the suggestion would have made things worse. BGFX_DISCARD_NONE-style uniform preservation captures the [m_uniformBegin, m_uniformEnd] range per draw and the kept range grows unboundedly, producing O(N²) GPU-side uniform processing for N draws. The actual win came from the Encoder route, which is a different mechanism entirely. The audit note (#R1) preserves the discovery so future engineers don't redo the same investigation.

Instancing

InstancedMeshComponent groups entities by instanceGroupId; InstanceBufferBuildSystem walks the visible groups, calls bgfx::allocInstanceDataBuffer() for each, and submits one draw per group. Caps that don't support instancing fall back to non-instanced draws. Recent audit win (#instance-pmr-map): the group map switched from ankerl::unordered_dense::map on the default heap to the pmr::map alias backed by the frame arena — measured 1.95× faster (1973 ns → 998 ns per frame at 8 groups × 32 instances).

5b. Animation

The 30-second answer. glTF clip loading via cgltf, keyframe sampling with Vec3 mix + Quat slerp, pose blending, CCD and FABRIK IK solvers. The skeleton uses a hot/cold split — parentIndices in one contiguous array, inverseBindMatrices in a second, nameHashes in a third — so the parent walk touches ~20× less cache than the old AoS layout.

Data structures: Skeleton with parallel arrays (parentIndices, inverseBindMatrices, nameHashes), AnimationClip with sorted keyframe channels per joint, Pose as an array of TRS JointPoses parallel to the skeleton. Animation events are sorted by time, fired in window.

Bone matrix composition is a single forward pass — joints are topologically sorted so parent always precedes child:

worldTransforms[i] = worldTransforms[parent[i]] * localTRS(pose[i]);
boneMatrix[i]      = worldTransforms[i] * inverseBindMatrices[i];

Output is appended to a shared per-frame bone buffer allocated from the FrameArena, with SkinComponent::boneMatrixOffset + boneCount recording each entity's window.

The audit landed several wins here:

  • #M1 Skeleton hot/cold split — the headline win. The old Joint struct was 80 B AoS (full inverseBindMatrix + parent + hash + padding); the parent walk read 4 B of that per joint. Now three parallel arrays: parent walk touches ~4 cache lines per 64-joint character instead of ~80.
  • #anim-pose-by-value — PoseComponent::pose is now Pose by value, so the InlinedVector<JointPose, 128> inside reuses its inline / heap buffer frame-over-frame instead of re-allocating from the arena.
  • #anim-trs — per-joint local matrix construction rewritten from translate * mat4_cast(quat) * scale (~176 muls) to direct TRS via mat3_cast(q) * diag(s) (~29 muls).
  • #anim-worldT — reserve + emplace_back instead of value-initialising the bone buffer to identity then overwriting.
Animation wins that DIDN'T pan out (be honest about these)

Two items got investigated and rejected ([skip] in the audit):

  • Skip rest-pose init for joints touched by channels (AnimationSampler.cpp:79-93). Microbench showed +6% to +10% regression because the rest-pose copy runs at ~0.7 ns/joint out of L1 cache; the bitmask construction + restPose-as-default loads inside the channel-sampling loop added more work than they saved.
  • glm::slerp short-circuit on equal keyframes. No measurable improvement: glm::slerp already internally checks dot(a, b) > 0.9995 and falls to nlerp, so the equal-quats case was already cheap.

The audit projections were wrong about the workload being memory-bandwidth bound. The microbenchmark is preserved at tests/animation/TestAnimationSampler.cpp with [anim-bench] for future re-evaluation.

5c. Physics

The 30-second answer. Jolt Physics behind an IPhysicsEngine interface — Jolt is multi-threaded, mature, and ships with Horizon Zero Dawn. The abstraction lets a NullPhysicsEngine swap in for headless tests. Box / sphere / capsule / mesh shapes; Static / Kinematic / Dynamic body types. The system fixes a 1/60 s timestep with the accumulator pattern (max 4 substeps). The nontrivial bit is hierarchy-aware sync: parented bodies are converted between world and local space using a custom math::inverseTRS() helper that's 3.4× faster than glm::inverse.

Components:

  • RigidBodyComponent (28 B) — Jolt body ID, mass, damping, friction, restitution, type (Static / Kinematic / Dynamic), collision layer.
  • ColliderComponent (32 B) — shape (Box / Sphere / Capsule / Mesh), offset, half-extents, radius.
  • PhysicsBodyCreatedTag — zero-size tag, set once the Jolt body exists.

Frame flow:

  1. Register new bodies: entities with RigidBody + Collider but no PhysicsBodyCreatedTag get a Jolt body created from their world-space transform.
  2. Sync kinematic bodies: read WorldTransformComponent, decompose, call physics.moveKinematic().
  3. Step physics: physics.step(deltaTime, kMaxSubSteps=4) — Jolt runs broadphase + narrowphase + constraint solve on its internal thread pool.
  4. Write back dynamic bodies: read new pose from Jolt, convert world → local if parented (using math::inverseTRS()), write TransformComponent, set the dirty flag so TransformSystem picks it up next frame.

The inverseTRS story (audit #phys-inverse-trs) is a nice talking point. For M = [R*S | t] with R orthonormal and S diagonal, the inverse is computable in ~30 muls without cofactor expansion: per-column |M.col_i|² = s_i², then A⁻¹[i,j] = M[j,i] / s_i² and t' = -A⁻¹ * t. glm::inverse does general 4×4 cofactor (~100 muls). Measured 3.4× speedup: median 3.0 ns vs 10.2 ns per call on M-series Mac, with the speedup growing on slower ARM little cores.

5d. Audio

The 30-second answer. SoLoud is the audio engine (mixing, voices, 3D positional, distance attenuation, Doppler, fading), miniaudio is the backend abstraction (CoreAudio on macOS/iOS, AAudio on Android with OpenSL ES fallback, WASAPI on Windows). IAudioEngine abstraction makes a NullAudioEngine swap in for tests. The interesting war story is the AAudio gate race — SoLoud's init sequence meant AAudio could fire its first callback before SoLoud finished allocating its resample buffers, causing a 100%-reproducible crash on Pixel 9. I fixed it with an atomic publish/acquire gate.

SoLoud chosen over FMOD primarily for binary size (~1.5 MB vs 3–6 MB) and licensing (FMOD has royalty terms above $200k revenue). FMOD is documented as the upgrade path for AAA polish (HRTF, geometry-aware reverb, occlusion, sound designer workflow).

3D positional audio uses two components: AudioListenerComponent (typically on the camera) and AudioSourceComponent (on any sound-emitting entity). AudioSystem reads WorldTransformComponent for both — no separate audio transform. Forward/up vectors are extracted from columns 2/1 of the world matrix.

War story: the AAudio gate race

Documented at docs/NOTES.md:2785-2867. The problem: SoLoud's Soloud::init calls in this order:

ma_device_init(NULL, &config, &gDevice);  // AAudio callback thread starts HERE
aSoloud->postinit_internal(...);          // allocates mResampleData[] HERE
ma_device_start(&gDevice);               // intended-but-irrelevant start point

On Pixel 9 / Android 16, AAudio's callback fires within microseconds of ma_device_init returning — before postinit_internal allocates the resample arrays. First callback calls Soloud::mix → mapResampleBuffers_internal → null-deref at +120 (mResampleDataOwner[i]). 100% reproducible. OpenSL ES doesn't have this behaviour (it waits for ma_device_start), which is why a previous "force OpenSL ES" workaround stopped the crash — at the cost of ~30 ms latency vs AAudio's ~10 ms.

The fix: a vendored third_party/soloud_patches/soloud_miniaudio.cpp wraps the SoLoud-mix callback in an std::atomic<bool> gAudioReady gate. Store-release after postinit_internal returns, acquire-load on every callback. While the flag is false, the callback memsets the output buffer to zero and returns.

Why this works: release/acquire ordering guarantees that every prior store in postinit_internal is visible to any callback that observes gAudioReady = true. Standard one-shot publisher/consumer pattern. Cost: one atomic load per audio callback (~1 ns on ARM with ldar). The silence-output window is the few ms between ma_device_init and postinit_internal — inaudible at startup, no sounds are loaded that early.

Bonus: backend-agnostic. Runs on every backend, so if a future miniaudio backend ever pre-starts its callback the same way (current candidate: WASAPI exclusive-mode), the same code catches it without a per-backend #ifdef.

Why not fix this in SoLoud upstream? Cleanest "real" fix would be reordering Soloud::init so postinit_internal runs before backend init. Requires patching SoLoud's 2k-line soloud.cpp (a vendored fork forever) or upstreaming to a maintainer-quiet project. The wrapper-side gate is small, contained, and in a file we already own.

5e. Assets

The 30-second answer. Handle-based typed asset table (AssetHandle<T> = 8-byte index+generation pair, phantom T for compile-time safety). Load is two-phase: worker threads do file I/O + CPU decode and push to a lock-free queue; the main thread drains the queue once per frame via assets.processUploads() and creates the bgfx resources (which must be main-thread on most backends). glTF via cgltf, KTX2 textures, ASTC on mobile, BC7 on desktop.

The constraint that shapes everything: bgfx does not allow GPU resource creation from worker threads. So I/O and CPU decode parallelise; GPU upload stays on the main thread.

Lock-free MPSC upload queue connects them. Game code polls state(handle) each frame; once Ready, the asset is usable.

Tier-aware asset resolver (engine/assets/TierAssetResolver.cpp) lets the same content ship at low / mid / high quality. The resolver falls back from low/foo.ktx to foo.ktx when a tier-specific variant isn't present, so games can selectively author per-tier assets without duplicating everything.

Compound assets: a .gltf file produces one GltfAsset with sub-handles per mesh, texture, material. Game code references the sub-handles. Editor hot-reload via FileWatcher is stripped from shipping builds via #ifdef ENGINE_EDITOR.

5f. Memory

The 30-second answer. Per-frame FrameArena is a std::pmr::monotonic_buffer_resource wrapped in a TrackingResource for HUD diagnostics. O(1) bump allocate, O(1) reset (just move the pointer back to the buffer start; no destructors). InlinedVector<T, N> gives SBO storage for small vectors. C++20 PMR was the deliberate choice over a custom arena because it interops with std::pmr::vector, the ankerl::unordered_dense::pmr::map, etc.

Default arena size: 2 MB (EngineDesc::frameArenaSize). Per-frame consumers include the animation bone buffer (16 KB for 256 joints; lazy-reserved so empty-view scenes pay zero) and the InstanceBufferBuildSystem group map.

Memory-layout-awareness is a recurring theme:

  • Components are laid out with explicit padding and static_assert on sizeof and offsetof. WorldTransformComponent is exactly 64 bytes / 16-byte aligned — one cache line per entity's world matrix.
  • The SparseSet dense storage is the textbook example — packed array → cache prefetcher-friendly.
  • Skeleton hot/cold split (#M1) was the big audit win in this category.

5g. Threading

The 30-second answer. Custom ThreadPool v2 with a POD submitTask(fn, arg) path (no std::function allocation), a bounded ring buffer (not std::deque), std::counting_semaphore for work-available wakeup, and an atomic activeTasks_ counter that notifies only on the transition-to-zero edge. The compile-time SystemExecutor dispatches phases through that POD path. Single-threaded by default; opt-in via EngineDesc::useSystemThreadPool because the per-system Reads/Writes declarations aren't on the built-in systems yet.

The ThreadPool v2 rewrite (audit #H1) was three changes in one:

  • Storage half: was std::deque<std::function<void()>> — heap allocation per submit when the closure exceeded SBO. Now std::array<Task, 1024> ring where Task = {void(*)(void*); void*} is 16-byte POD. submit(std::function) kept as back-compat wrapper that heap-allocates a std::function* and dispatches via submitTask with a free-on-run trampoline.
  • Wakeup half: workers waiting on workCv_ with the queue mutex held serialised wakeups. Now std::counting_semaphore<INT32_MAX> for work-available — submitTask does workSem_.release() outside the ring mutex; workers do workSem_.acquire() without holding any other lock.
  • Notify half: was --activeTasks_ inside lock then unconditional doneCv_.notify_one(). Now atomic activeTasks_.fetch_sub(1, acq_rel); doneCv_.notify_all() fires only when the returned prior value is 1.

SystemExecutor drives phases through this POD path. 7 unit tests pin behaviour including a 10000-frame conservation race-check that's TSAN-friendly.

The "DAG scheduler" exists as scaffolding but isn't yet steering the built-in systems. EngineDesc::useSystemThreadPool defaults to false so historical single-threaded behaviour is preserved. The per-system Reads/Writes declarations on TransformSystem, FrustumCullSystem, DrawCallBuildSystem, etc. are deferred to per-tier device measurement. Still TODO: per-worker queues + work-stealing for true MPMC scaling — the current ring uses one short-held mutex.

6. Editor

The 30-second answer. sama_editor is a panel-based editor (Hierarchy, Properties, Console, Viewport) that embeds the engine's bgfx viewport. The design goes native on each platform — AppKit on macOS, Win32 on Windows — rather than Qt or ImGui-everywhere, because native gives better text rendering and system integration (drag-drop, file dialogs, dark mode) without a 200 MB Qt dependency. 7. Scene authoring & persistence

The 30-second answer. Scenes are authored interactively in the editor and saved as human-readable JSON via SceneSerializer — a registry of named per-component (serialize, deserialize) handler pairs. The same serializer drives both editor save/load and runtime loads. The runtime adds SceneManager for lifecycle (load / unload / reload, persistent entities that survive scene transitions). Project-level configuration (window, tier, physics, audio defaults) lives separately in project.json parsed by ProjectConfig.

Design notes:

  • Human-readable on purpose. An artist can diff two scene files in a code review and see exactly what changed. Binary formats would be smaller but opaque.
  • Scene-local IDs (not runtime EntityIDs) — the runtime ID has a generation field that changes across loads, and the same scene loaded twice should produce the same parent → child topology even though the underlying entity slots are different. The loader builds a localId → EntityID map and resolves the parent reference in a second pass. See engine/scene/SceneSerializer.cpp:520-590.
  • Mesh / Material as integer handles into RenderResources, not paths. This decouples the scene format from asset paths and lets the editor track an unsaved scene against unsaved-asset state. Downside: the same scene loaded with a different RenderResources would reference the wrong meshes — out-of-scope for the current single-project assumption but noted as a future hardening target.
  • Unknown components are silently ignored on load — forward-compatibility for game code that registered its own components against a scene saved by an older build.

Follow-up: why JSON and not a binary format?

Three reasons. (1) The format is small enough that the parse cost is invisible — rapidjson handles a typical level in <5 ms. (2) Diff-ability in code review: artists and gameplay engineers can both read it. (3) The editor saves often; readable failures (line + column in the error) beat opaque "corrupt save." The cost is file size — a binary format would be ~3× smaller for the same scene. If we ever hit the size ceiling, the migration path is documented: switch the writer to write CBOR while keeping the same key names; loader auto-detects via the leading byte.

Follow-up: what about SceneManager at runtime?

The editor uses SceneSerializer directly because it wants fine-grained control over the registry (undo stack, dirty panels). Games use the higher-level SceneManager wrapper (engine/scene/SceneManager.{h,cpp}) which adds:

  • Active-scene tracking via SceneHandle — one int per loaded scene; the manager remembers which entities belong to which scene file.
  • Persistent entities — call sceneManager.markPersistent(player) and that entity survives the next unloadScene() / loadScene("level2.json") pair. Useful for the player avatar, inventory, persistent UI.
  • Reload for dev iteration — reloadScene() does unload + load(samePath) so a designer can save in the editor and immediately re-run the game without restarting the process.
  • Onload callback for gameplay code to wire up references after entities exist (sceneManager.setOnSceneLoaded([](SceneHandle h){ … })).
Follow-up: what about glTF — that's a "scene" too, isn't it?

glTF is treated as an asset, not a scene. engine/assets/GltfSceneSpawner.cpp imports a glTF model into the running registry (one entity per glTF node, with Transform / Mesh / Material populated). The result can then be saved into the engine's .json scene format like anything else. glTF is the "DCC bridge"; the JSON scene is the engine's native authored layer that sits on top. Tests in tests/scene/TestSceneSerializer.cpp cover round-trip fidelity for the built-in components.

Honesty flags

  • The Mesh / Material handles are not portable across RenderResources instances yet. In practice that means a scene file is implicitly tied to the project that saved it; loading it into a different project may bind the wrong meshes silently. A future hardening pass would store stable asset names (or content hashes) and resolve at load.
  • The "unknown component" silent-ignore is good for forward-compat but means a typo in a registered component name fails open — the data is silently dropped on load. A validation pass that diffs registered names against the file's keys would catch this.
  • The scene-local ID assignment is sequential by iteration order through Registry::forEachEntity. Saving the same scene twice can produce different IDs if entity creation order shifts. Doesn't affect correctness (the parent references are rewritten consistently inside the file) but does produce noisy diffs. A future fix: sort entities by name before ID assignment.
  • The serializer is not thread-safe — runs on whichever thread called save/load. Editor / runtime both call from the main thread, so this hasn't bitten yet.

11. Known limitations & what I'd do differently

Better to lead with these than be caught off guard:

  • The DAG scheduler is scaffolding-only for now. Built-in systems don't declare Reads/Writes; useSystemThreadPool defaults to false. Real frame parallelism is per-game opt-in until the per-tier device measurement happens.
  • ThreadPool is single-mutex MPMC, not work-stealing. Per-worker queues + work-stealing would help under heavy submit contention.
  • GLTF texture ownership refactor. UiRenderSystem::update still takes bgfx::ProgramHandle + bgfx::UniformHandle directly — the bgfx-free wrapper layer hasn't closed that boundary yet.
  • Tier-aware shader variants are stubs. BGFX_CONFIG_MAX_BONES=128 applies even on low-tier (maxBones=64); the deferred LOW_TIER PBR variant that would skip the cluster loop on no-lights scenes is open (#S1 texture-indirection half).
  • Surface.setFrameRate plumbing on Android is missing. TierConfig::targetFPS=30 is honoured on iOS but not Android — the panel will still drive 60 Hz vsync and the engine produces extra frames that don't matter (#P2).
  • No cascade-transition blending in CSM.
  • Audio is single-listener. Multi-listener (split-screen) is unimplemented; the priority field exists but only the highest wins.
  • No render graph. Passes are a hardcoded linear pipeline. A proper RG is the natural next step beyond ~6 passes — noted in docs/RENDERING_ARCHITECTURE.md as a future upgrade.

Deliberate exclusions (mention these so the interviewer knows scope was considered)

  • Networking / multiplayer — intentionally not in scope. Single-player engine. libcurl is wired for asset/update downloads only.
  • VR / AR — out of scope.
  • Localisation — out of scope.
  • Scripting layer — the IGame pattern + C++ gameplay is the model. No Lua, no C#, no embedded VM.
  • Ray tracing — explicitly tabled long-term in the initial proposal. bgfx doesn't expose RT APIs; forking bgfx would be a larger project than the engine itself.
  • Logging strategy — currently just bgfx's callback + per-platform __android_log_print / NSLog; no structured logging library. Acceptable scope decision.

12. Likely interview questions & ready-to-use answers

"Why bgfx and not your own abstraction?"

The binary-size win was ~1–2 MB; the cost would be months of VkInstance+swapchain+pipeline-cache boilerplate per backend. As a solo developer on a multi-platform engine, that's months not spent on the rendering techniques I actually wanted to build. bgfx gives me one shader source → MSL/SPIR-V/HLSL via shaderc, the Encoder API for multi-threaded draw recording, and a render thread by default so bgfx::frame() is an async hand-off. MIT-licensed, mature on mobile.

"Walk me through a frame from input to pixel."

GLFW (or NativeActivity / UIApplication on mobile) hands events to InputSystem, which diffs against last frame to produce pressed/released state in InputState. Game tick runs animation (sample clips, blend, compose bones into a per-frame arena buffer), then physics fixed-step (1/60s, up to 4 substeps via Jolt), then TransformSystem recursively rebuilds dirty world matrices. Visibility pass: FrustumCullSystem sets VisibleTag, ShadowCullSystem sets ShadowVisibleTag cascade bits, LightClusterBuilder fills the 16×9×24 grid into three GPU textures. Render pass: DrawCallBuildSystem opens a bgfx::Encoder, walks the visible entities, records draw calls per shadow cascade and the opaque pass. Post-process: SSAO, bloom (2N-1 passes), ACES tonemap, FXAA. Then endFrame() resets the FrameArena and calls bgfx::frame() which hands off to bgfx's render thread for the actual GPU submit.

"How does forward clustered lighting compare to deferred?"

For mobile and TBDR specifically, clustered forward wins. A G-buffer is 4–5 fullscreen render targets, and on TBDR every store flushes tile memory to main RAM — that's the actual bandwidth bottleneck. Forward+ keeps shading in-tile. Transparency works naturally (no parallel forward path needed). Memory footprint is lower. The cost is shading the same fragment more times under overdraw, but with depth prepass disabled on TBDR (also for bandwidth reasons) and decent occlusion culling, overdraw isn't catastrophic. For a desktop AAA renderer at 4K with hundreds of lights and complex post-process chains, deferred or visibility buffer would be a different conversation.

"What's the performance bottleneck on low-tier Android?"

Fragment shader cost and main-memory bandwidth, both fed by Mali-G57's modest ALU throughput. Two specific paths: the clustered light loop in fs_pbr.sc samples three RGBA32F textures per light, and tonemap's pow(x, 1/2.2) chokes Mali. Tier-aware fixes addressed by the audit: cap bloom downsample steps to 0 on low (off entirely), force depth prepass off (TBDR), mediump on BRDF / sample paths for fp16, and target 30 fps to give the SoC thermal headroom. Open items I'd land next: pow → sqrt in tonemap, setTexture batching for frame-constant slots, and a LOW_TIER PBR variant that skips the cluster loop for no-lights scenes.

"How do you avoid races in ECS access?"

Three layers. First, SparseSet::get / contains are pure reads — concurrent reads are safe by construction as long as no one mutates the registry. Second, structural mutations (emplace, remove, destroyEntity) are NOT safe; the design is to defer them through a CommandBuffer and play back on the main thread after a phase barrier. Third, the compile-time conflict detection in buildSchedule<...>() places systems that write the same component type in serial phases, so even if two systems would only ever touch disjoint entity sets, the type-level rule keeps them safe by being conservative.

"What was the hardest bug you fixed?"

The AAudio race in SoLoud's init. 100% reproducible SIGSEGV on Pixel 9 in mapResampleBuffers_internal. The problem was that SoLoud calls ma_device_init before postinit_internal, but on Android 16 the AAudio backend starts its callback thread inside ma_device_init and the first callback fires before SoLoud allocates the resample arrays. OpenSL ES doesn't have this behaviour, which is why "force OpenSL ES" stopped the crash — at the cost of 30 ms latency vs AAudio's 10 ms. I fixed it with an atomic publish/acquire gate in our vendored soloud_miniaudio.cpp: store-release after postinit_internal returns, acquire-load on every callback. While the flag is false, the callback memsets the output buffer to zero. Costs one atomic load per callback (~1 ns with ldar on ARM). The release/acquire ordering guarantees every prior store in postinit_internal is visible to any callback that observes true. Bonus: it's backend-agnostic, so the next miniaudio backend that pre-starts its callback (likely WASAPI exclusive-mode) gets the same protection without a #ifdef.

"How would you scale this to 10,000 entities?"

The frame-anatomy estimate puts a 10k-entity scene at ~10 ms on a single thread — over budget. The plan is the work that's already scaffolded: turn on the SystemExecutor with the per-system Reads/Writes declarations, run FrustumCull + ShadowCull + LightCluster in parallel via deferred command buffers, run AnimationSystem in parallel with AudioSystem on the simulation phase, and split DrawCallBuildSystem's entity list across worker Encoders. Estimated saving ~3.5 ms — gets it back under 60 fps budget. Memory-side: the LightClusterBuilder fixed arrays (~150 KB always-resident) should become tier-aware (low → 8×4×8 grid instead of 16×9×24).

"Why ECS over an OO scene graph?"

Cache locality is the practical answer. Iterating 50k Velocity components linearly through a packed dense array is what the prefetcher expects; chasing virtual-dispatch GameObject* through a tree isn't. The structural answer is composition: a "player" isn't a kind of anything, it's has-a transform + mesh + animator + rigidbody + audio listener — each owned by a different subsystem. ECS expresses that natively. Note though: the engine does still have a hierarchy via HierarchyComponent + ChildrenComponent and a TransformSystem that does top-down recursive composition. "ECS-as-scene-graph" — the hierarchy is encoded directly in ECS components, no separate scene graph object. Best of both.
